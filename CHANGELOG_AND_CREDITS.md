# Vibe CV Copilot 迭代记录与鸣谢 (Changelog & Credits)

本项目从最初的一个简单排版脚本，逐步进化为今天的“一条龙简历智能体”，离不开开源社区里许多优秀前辈的探索与沉淀。本文件用于记录我们如何站在巨人的肩膀上，打造这套属于我们自己的**简历自动生成与打磨 AI Agent Skill** 的更新历程。

## v0.1: 引入基础工作流框架 (Li-Baichuan-James/cv-skill)
**日期**: 2026-04-13
**来源**: [Li-Baichuan-James/cv-skill](https://github.com/Li-Baichuan-James/cv-skill)

### 1. 这个基础框架能实现什么？
这是一个非常成熟的基于 Agent 工作流的简历制作系统，核心亮点在于**“拒绝 AI 幻觉，坚持事实优先”**以及**“通过 LaTeX 输出所见即所得的 PDF”**。
- **全自动四步工作流**：它将制作简历拆分为了 4 个子技能：
  1. `resume-intake-and-extraction` (信息读取与提取)
  2. `resume-crafter` (主入口，负责调度)
  3. `resume-authoring-and-assembly` (起草和排版)
  4. `resume-review-and-delivery` (事实复核与交付)
- **事实追踪 (Claim-Source Map)**：在撰写前，它会强制要求 AI 生成 `claim-source-map.md`，把简历里的每一句话和用户的聊天记录/源材料对应起来。如果遇到不确定的点，它会主动停下来问用户，绝不脑补瞎编。
- **内置 LaTeX 模板**：自带了 `中文标准模板`、`英文学术模板 (Research)`、`英文求职模板 (Industry ATS)` 等，直接输出 `.tex` 和 `.pdf` 文件，彻底解决网页版 AI 格式错乱的问题。

### 2. 为什么它非常适合作为我们的基础？
- **架构极度清晰**：它的输入输出格式、边界定义（适合做什么/不适合做什么）非常严谨。
- **可以直接拿来魔改**：它自带了现成的 LaTeX 模板和编译逻辑，我们不需要从零去写排版代码。
- **我们后续的整合计划（待办）**：
  - [ ] 依托我们的具体项目（如 Crypto-Behavioral-Bias-Benchmark），写一个“自动将指定 Markdown 论文/项目报告转化为 STAR 法则简历”的专属 Prompt。
  - [ ] 将我们自己的审美/行业特定的黑话字典融入到它的 `resume-authoring-and-assembly` 环节中。
  - [ ] 简化它的多技能依赖，或者用一个超级入口把这些功能统合，让 VibeCoding 的体验更流畅。

## v0.2: 引入 JD 匹配与项目素材库理念 (spontaneousai/job-hunt-copilot)
**日期**: 2026-04-13
**来源**: [spontaneousai/job-hunt-copilot](https://github.com/spontaneousai/job-hunt-copilot)

### 1. 这个仓库补充了什么核心能力？
如果说 v0.1 的 `cv-skill` 解决的是“怎么把简历排版好、怎么保证不出错（PDF+LaTeX）”，那么 v0.2 的 `job-hunt-copilot` 解决的就是**“怎么针对不同的 JD 投其所好”**。
它的核心理念是建立一个**「项目素材库（Material Library）」**：
- **解耦经历与输出**：你把做过的所有项目（哪怕很杂乱）存进库里。当你要投递某个具体岗位时，把 JD 发给 AI，AI 会自动从素材库中“挑选”最匹配的项目，并用 JD 里的关键词（Keyword Matching）去重构项目的 STAR 叙事。
- **面试讲稿与模拟面试**：它不仅改简历，还能根据 JD 预测面试官的追问，直接生成中英双语的口述讲稿，并提供 Mock Interview 功能。

### 2. 我们的终极 VibeCoding 整合方案 (v0.1 + v0.2)
这两个项目简直是天作之合。我们接下来的自建 Skill 将融合两者的最强能力，打造一个**从 JD 到最终 PDF 简历的一条龙 Agent**：

1. **输入阶段 (借鉴 v0.2)**：
   - 用户输入目标岗位的 JD。
   - Agent 读取我们本地的 Markdown 资料库（比如刚做完的 `Crypto-Behavioral-Bias-Benchmark` 项目报告）。
2. **加工阶段 (借鉴 v0.2 叙事重构)**：
   - Agent 自动提取 JD 的核心能力要求（如数据分析、大模型应用、量化交易）。
   - Agent 用这些关键词重写项目的 Bullet Points（STAR 法则），强调我们项目中匹配的特质。
3. **输出阶段 (借鉴 v0.1 事实校验与排版)**：
   - Agent 生成 `claim-source-map.md` 进行防幻觉核查。
   - Agent 将重写好的内容填入 LaTeX 模板（比如 `research/ats` 或 `industry/ats`）。
   - 直接编译输出排版完美的 PDF 文件供我们投递。
   - 顺便生成一份针对该 JD 的面试口述讲稿。

## v0.3: 引入专业猎头级 7 步深度重构流 (Deep Structured Pipeline)
**日期**: 2026-04-13
**来源**: 社区优秀实践 (Workflow 深度结构化思路)

### 1. 为什么我们需要这 7 步？
之前我们的 v0.1 和 v0.2 解决了“怎么排版不出错”和“怎么建立素材库”，但它们依然依赖于大模型原生的“一次性对话直觉”。而这个新的思路，彻底把 AI 变成了**“专业猎头 + 简历咨询师”**：
- **拒绝表面润色**：引入统一 Schema，强制了解转岗背景、不可改字段等隐藏上下文。
- **深度意图解析**：不只是提取 JD 关键词，而是推理 HR 的“隐含偏好”和“业务语境”。
- **先评估再动笔**：在生成前增加 Alignment Layer（匹配评分），明确“差距在哪”；在生成后增加 Validation Layer（风险校验），杜绝“浓浓的 AI 味”和“强行贴靠 JD 的生硬感”。

### 2. 最终 Vibe CV Copilot 的完全体工作流
我们将把这个 7 步流作为核心 Prompt 写入 `vibe-cv-copilot` 的大脑（`SKILL.md`）中：
1. **输入标准化 (Schema Intake)**：收取 JD、行业、转岗背景等全量上下文。
2. **JD 解析 (Intent Inference)**：拆解核心职责、必备能力、隐含偏好与 ATS 词库。
3. **简历结构化 (Evidence Tagging)**：把 `resources/` 里的素材切分为能力单元，并打上“业务场景/量化证据”标签。
4. **匹配评分 (Gap Analysis)**：输出 JD 与素材的 Must-have 匹配度、风险分，确定修改策略。
5. **受约束生成 (Constrained Generation)**：仅基于事实进行重组与表达优化，坚守防幻觉底线。
6. **风险校验 (Validation Layer)**：自查 AI 腔调、夸张表述和模板感。
7. **多模态输出 (Document & PDF)**：输出 LaTeX 排版的 PDF，并附带定制说明、前后 Diff 对比以及面试讲稿。

## v0.4: 引入“决策前置”的轻量级快筛模式 (Quick Assessment Mode)
**日期**: 2026-04-13
**来源**: 社区插件产品思路提取

### 1. 这个思路解决了什么盲区？
我们之前的 7 步重构流（v0.3）是一个“重武器”——一旦决定要投，就开启全套的精细化重构和 LaTeX 排版。但这忽略了求职者的一个极其高频的真实场景：**“刷岗时的快速决策”**。
- 用户在看 JD 时，很多时候不是要立刻改简历，而是要决定：**“这岗位值得投吗？我投的话胜算大吗？需要大改还是小改？”**
- 如果每次看 JD 都要跑一次完整的 7 步流，不仅耗时，而且心理负担极大。

### 2. 我们如何将“插件思维”融入 Skill？
我们不需要把它做成独立的浏览器插件，而是要在我们的 `vibe-cv-copilot` 中增加一个**前置的轻量级模式（Mode: Quick Assess）**。

当用户开启这个模式并扔入多个 JD 时，Agent 不会直接去改写简历，而是充当“副驾扫描仪”：
1. **一秒拆解 JD**：提炼该岗位的核心任务、能力要求和关键词。
2. **轻量级“照镜子”提示**：快速对比素材库，只给出方向性建议（例如：“你的项目里有活动执行，但转化结果不明显”、“你摘要里没体现这个岗位要的方向”）。
3. **投递价值判定**：给出一个“大改/小改/不值得投”的快速判定结论。

通过这种方式，我们的 Skill 既有帮助用户“批量做决策”的轻盈感，又有在决策后一键启动 7 步重构流的重火力支撑。

## v0.5: 引入行业认知与职业知识库管理 (Career Knowledge Base)
**日期**: 2026-05-16
**来源**: 用户实际求职探索需求

### 1. 这个思路解决了什么问题？
在找实习或校招的过程中，候选人（尤其是低年级同学）往往在探索不同的方向。日常会听讲座、看小红书/知乎笔记、收集各种岗位的面经和认知。如果这些碎片化信息没有沉淀，面试时就无法调取。

### 2. 我们如何在工作流中实现？
我们新建了一个专门的目录 `Career_Knowledge_Base/`。
- **信息自动拆解与沉淀**：当用户丢来一段杂乱的讲座笔记或社媒链接时，Agent 会自动识别所属的“岗位”或“行业”。
- **结构化归档**：将零散笔记整理为“核心能力、面试策略、入行路径、高频 Q&A”等结构化模块，补充到对应的 `.md` 认知库中（如 `Product_Manager.md`）。
- **投递与面试联动**：未来在生成简历和面试讲稿（Pitch Script）时，Agent 可以直接从这些认知库中提取“行业黑话”和“业务痛点”，让内容更显专业和真诚。

## v0.6: 彻底攻克中文 LaTeX 排版与单页限制 (Custom Chinese LaTeX Template)
**日期**: 2026-05-16
**来源**: 针对用户提供的 Word 范例进行极限调整

### 1. 为什么原版排版“水土不服”？
原版 `cv-skill` 的 LaTeX 模板（`resume.cls`）是基于纯英文环境（10pt, 标准行距, 较大的 section 间距）设计的。当直接用于中文时，暴露了几个致命问题：
- **字体加粗乱码**：默认 SimSun/KaiTi 没有自带的粗体字重，导致 `\textbf` 失效或报错。
- **空间极度浪费**：英文排版追求留白（White Space），但中文简历（尤其是求职实习）讲究“一页纸高密度信息”，原模板的段落间距、列表上下间距、标题与正文间距过大，导致内容轻易溢出到第二页。
- **元素重叠与空行失控**：使用 `\\` 强制换行配合 `itemize` 列表时，由于 LaTeX 的段落机制，会导致莫名其妙的额外空行；而单纯压缩负间距（`\vspace`）又会导致标题文字和横线重叠。

### 2. 我们自己的“中文单页标准模板”改造
通过对 `resume.cls` 和 `resume.tex` 的多次极限调优，我们形成了适合国人 HR 审阅习惯的单页模板：
- **引入伪粗体**：使用 `\setCJKmainfont[AutoFakeBold=true]{SimSun}` 强行渲染中文粗体，完美解决乱码问题。
- **彻底消除空行**：废弃 `\\` 换行，改用 `{内容}\par\vspace{-0.5em}`，从底层控制段落间隔，防止 `itemize` 产生不必要的空行。
- **毫米级极限压缩**：
  - 页边距缩小至 `0.05in` (配合 `extarticle` 的 `9pt` 字体)。
  - 行距 `\linespread` 压至 `0.8`。
  - 列表内部距 `itemsep` 缩至 `-0.2em`。
- **优雅的版面合并**：在内容极其丰富时，Agent 会智能地将“校园经历”和“荣誉奖项”合并为一个 Section，从而节约一个大标题的纵向空间，最终实现严丝合缝的 **Perfect 1-Page PDF**。

## v0.7: 引入 HR 总监视角与动态技能拆解 (Dynamic XYZ/STAR & Verbatim Retention)
**日期**: 2026-05-16
**来源**: 社区高阶 HR Prompt 经验与用户真实修改痛点

### 1. 为什么需要保留原版范例 (Verbatim Retention)？
在之前的版本中，Agent 每次都会把用户的整个简历“重写”一遍。但这会带来一个痛点：像“校园经历”、“荣誉奖项”、“专业技能”这些部分，用户自己其实已经手工打磨到了最佳状态（字斟句酌）。如果不加限制，AI 往往会“画蛇添足”地修改这些不需要根据 JD 变动的通用信息。
- **解决方案**：引入 `样板范例.md/docx` 概念。在 `vibe-cv-copilot` 中新增规则：对于非核心经历部分，Agent 必须**一字不差地照抄样板范例**，只将算力集中在“实习”和“项目”这些需要匹配 JD 的核心板块上。

### 2. HR 总监视角与动态“四字小标题”
用户反馈，优秀的简历在 bullet points 开头往往会有诸如“数据分析”、“模型优化”、“架构设计”这样的四字小标题，让 HR 一秒抓住重点。同时，需要强化 XYZ 法则（为了完成 X 任务，采取了 Y 方法，达成了 Z 结果）。
- **不用死记硬背的技能库**：与其我们自己维护一个庞大且死板的“行业四字技能词典”，不如利用大模型的强阅读理解能力。
- **动态抽取 (Dynamic Extraction)**：我们在工作流的 **Step 2 (JD 解析)** 中，新增了一个专门的子任务：让 Agent 直接根据当前的 JD 提取或生成一组“四字核心技能词”。
- **XYZ 约束输出**：在 **Step 5 (受约束生成)** 中，Agent 会直接用这组刚刚提炼出的四字词作为加粗小标题，并严格使用 XYZ/STAR 法则重构项目经历，呈现量化数据和专业技能。这让生成的简历极具“高级职场感”。

## v0.8: 放弃 LaTeX，全面转向 Docx 模板注入 (Docx Template Injection)
**日期**: 2026-05-16
**来源**: Trae 多模态文档处理的惊艳效果与用户反馈

### 1. 为什么放弃 LaTeX？
在之前的版本中，为了实现精准排版，我们耗费了大量精力调试 LaTeX 的底层模板（如解决中文字体加粗乱码、强行压缩页边距与行距）。但这带来了一个致命问题：LaTeX 对于大部分非计算机专业用户来说门槛过高，一旦需要手动微调（比如删掉半行字、加粗某个词），普通用户根本无法在 `.tex` 源码中快速操作。

### 2. 为什么转投 Docx 模板注入？
用户在实际测试中发现，直接让大模型基于 `排版范例.docx` 进行内容替换，输出的排版完美保留了原有的页边距、楷体/Times New Roman 字体、字号等所有手动设置。
- **底层原理 (XML 样式继承)**：Docx 本质上是一个包含 XML 标签的压缩包。只要我们通过 Python 脚本（如 `python-docx`）直接在底层的 `<w:t>` 文本节点进行替换，就能让新生成的文字完美继承原节点 `<w:rPr>` 里的所有样式（加粗、字号、中英文字体隔离），从而实现真正的“所见即所得”。

### 3. v0.8 核心工作流变更
- **不再生成 PDF/TeX**：Agent 的输出环节从繁琐的 `xelatex` 编译，变更为执行底层的 `scripts/generate_docx.py`。
- **工作流闭环**：
  1. 照常通过大模型进行 JD 解析与 XYZ 法则重写，生成 `Markdown` 版本。
  2. Agent 自动编写并执行替换脚本，将 Markdown 中的核心项目直接注入到 `resources/排版范例.docx` 中。
  3. 输出名为 `YYYYMMDD-公司岗位-简历-姓名.docx` 的最终文件。这不仅 100% 保留了视觉美感，还赋予了用户极大的**后续手动微调自由度**。

## v0.9: 引入人工在环审核机制 (Human-In-The-Loop Review)
**日期**: 2026-05-16
**来源**: 解决 AI 幻觉与用户精细化修改痛点

### 1. 为什么需要审核机制？
虽然大模型可以通过 XYZ/STAR 法则改写出极具职场感的文案，但直接将 AI 的输出注入最终的 Docx 文件存在两个风险：
- **AI 幻觉与强行贴靠**：大模型可能会为了迎合 JD，稍微夸大原有素材的某些事实。
- **缺乏手动修改空间**：由于 IDE（如 Trae）无法直接预览或编辑底层的二进制 `.docx` 文件，如果直接生成最终版，用户发现问题后再回去改就很麻烦。

### 2. Markdown 作为解耦层 (The Review Draft)
用户提出了一个极其专业的工程实践——**人工在环 (HITL)**。
- 我们在工作流的中间插入了一个全新的步骤：**Step 6: Human-In-The-Loop Review**。
- Agent 在重写完所有经历后，不会立刻生成 Docx。而是会先在 `work/` 目录下生成一个名为 `review_draft.md` 的中间件文件。
- 这个文件会以清晰的**对照视图**（原始素材 -> AI 改写草稿）展示所有变动。
- **暂停与接管**：Agent 会主动**暂停运行**，让用户在 IDE 里打开这个 Markdown 文件，像编辑普通代码一样进行手动的删改、润色。
- 当用户回复“确认无误”后，Agent 才会进入最后的 Docx 注入阶段。这种“AI 负责重构起草，人类负责拍板定稿”的模式，彻底打通了人机协同的最后一公里。

## v0.10: Review Draft 动态命名机制 (Dynamic Draft Naming)
**日期**: 2026-05-16
**来源**: 优化单文件覆盖问题

### 1. 为什么不能只用一个统一的 `review_draft.md`？
正如用户所指出的，在海投简历时，我们经常会针对不同的公司、不同的岗位同时生成好几份简历草稿。如果所有的审核文件都叫 `review_draft.md`，就会导致：
- 文件相互覆盖，无法保留历史记录。
- 难以追溯某一个特定的修改到底是针对哪个岗位的。

### 2. 动态命名策略
我们将 `SKILL.md` 中第 6 步生成的审核文件，强制加入了**“动态公司与岗位名”**的后缀约束。
- 格式要求：`review_draft_[Company]_[Role].md`
- 范例：`work/review_draft_TikTok_AI产品实习生.md`
- 这样不仅让所有的草稿独立留存，而且非常直观，一眼就能看出这是在改哪家公司的简历，完美契合了“一篇篇海投精改”的求职实战节奏。

## v0.11: 深度项目挂载与全知视角 (Deep Project Mastery)
**日期**: 2026-05-16
**来源**: 应对高强度面试 Mock 与数据真实性需求

### 1. 为什么 AI 不能只做“排版工”？
既然我们的 `vibe-cv-copilot` 是直接部署在本地项目代码库（如清深实习的 `Crypto-Behavioral-Bias-Benchmark`）同级目录下的，如果 AI 只负责改写文字而不去触碰底层的代码和数据，那是对 IDE 本地算力的巨大浪费。在面试官的追问下，仅凭几句漂亮的话是经不起深挖的。

### 2. 全知视角与代码重跑机制
我们在工作流中赋予了 AI 全新的权限和责任：
- **拒绝占位符糊弄**：当简历草稿中出现 `[XX]%` 等量化占位符时，AI 不应该让用户自己去“编”或者“找”，而是要主动提出去翻阅本地实验日志，甚至主动**重跑核心 Python 脚本**来获取真实的实验数据。
- **面试深度联防**：AI 必须对部署环境下的历史项目了如指掌。对于清深等核心项目，AI 会在 `Career_Knowledge_Base/Interview_Mock_QnA_*.md` 中自动列出“深度复盘 TODO”，包括清理废弃文件、串联逻辑、复现代码等，确保在生成简历的同时，帮用户把面试的“弹药库”填满。

## v0.12: 引入“无中生有”的项目孵化闭环 (0-to-1 Project Hatching)
**日期**: 2026-05-18
**来源**: [LiuMengxuan04/shushu-internship-tool](https://github.com/LiuMengxuan04/shushu-internship-tool.git)

### 1. 这个项目解决了什么终极痛点？
我们之前的 Vibe CV Copilot 无论是 `v0.1` 的排版还是 `v0.11` 的全知视角，都有一个硬前提：**用户必须已经做过一个相关项目**。
但对于大量“0 经验”或“项目完全不对口”的同学（即开源社区常说的“鼠鼠”），他们面临的困境是：“我连写在简历上的项目都没有，你排版改得再好有什么用？”
`shushu-internship-tool` 提出了一个天才般的逆向工程理念：**从 JD 反推开源项目，再把开源项目魔改成简历经历**。

### 2. 我们如何吸收它的核心能力（补全前置环节）？
我们将 `shushu-internship-tool` 的核心逻辑吸收为我们 Agent 的 **Phase 0 (项目孵化期)** 模块。当用户输入目标 JD 且本地素材库空白（或极度不匹配）时，Agent 将启动以下全新链路：
- **按图索骥 (Repo Discovery)**：直接从 GitHub 筛选出 2-3 个与 JD 技术栈高度匹配、有最小可运行路径 (Smoke Test)、且有改造空间的开源库。
- **摸底与审计 (Repo Audit)**：通过自动化脚本扫描开源库，生成核心代码路径、数据流和 API 入口，帮助用户在 1-2 天内极速“吃透”开源代码。
- **定制改造点 (Modification Playbook)**：为了防重合（防撞车），Agent 不会让用户直接抄开源库，而是针对 JD 规划高价值的“改造点”（例如：加个 Redis 缓存、引入微服务架构、换个大模型接口）。
- **闭环注入 (Interview Pack)**：将这套魔改后的项目成果，无缝对接到我们原有的 `vibe-cv-copilot` 工作流中，直接生成 STAR 法则经历、防幻觉的 Word 简历以及配套的面试 Q&A 拷问包。

通过引入这一步，我们的 Agent 真正实现了从“我连项目都没有”到“拿着精美简历去面试”的绝对“一条龙”闭环！

## v0.13: 引入沉浸式实习复盘与成果榨取 (Immersive Internship Retrospective)
**日期**: 2026-05-18
**来源**: [Sunanzhe2004/shushu-internship-resume-optimizer](https://github.com/Sunanzhe2004/shushu-internship-resume-optimizer.git)

### 1. 这个项目解决了什么痛点？
很多求职者在完成一段实习后，往往只留下了一些零散的代码仓库、工作周报或业务背景文档，很难系统性地将这些碎片转化为简历上极具竞争力的“高价值经历”。
`shushu-internship-resume-optimizer` 专门针对“已有实习/项目经验但不知道怎么写”的场景，提供了一套极致的**成果榨取工作流**。

### 2. 我们如何吸收它的核心能力？
我们将这个理念融入到 Agent 的 **复盘提取层 (Retrospective Layer)** 中。当用户提供自己过往实习的代码仓库或业务文档时，Agent 会执行以下操作：
- **深度代码与文档融合**：直接扫描用户的实习代码仓库、项目总结和业务背景材料，提取真实的技术栈、业务指标和工程难点。
- **降维提炼 STAR 草稿**：将冗长的业务逻辑和代码实现，转化为符合 HR 视角的 STAR 法则简历要点（Bullet Points），直接量化成果。
- **生成结构化面试复盘材料**：不仅改简历，还会根据提炼出的业务背景和技术难点，反向生成面试官可能追问的考点与应对策略，形成一套完整的“实习复盘武器库”。

有了这个模块，无论是“无中生有”找项目（v0.12），还是“榨取已有实习成果”（v0.13），我们的 One-Stop Resume Copilot 都能游刃有余地应对！
